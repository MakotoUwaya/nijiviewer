## セキュリティ評価

### ✅ 適切に実装されている項目

#### 1. WebAuthn 標準準拠

- **評価**: ✅ 良好
- **詳細**: Supabase の WebAuthn 実装は W3C WebAuthn 標準に準拠しています
- **根拠**:
  - `navigator.credentials.create()` / `navigator.credentials.get()` を使用
  - Challenge-Response 認証フロー
  - 公開鍵暗号方式（秘密鍵はデバイス内に保存）

#### 2. Relying Party (RP) 設定

- **評価**: ✅ 良好
- **詳細**: RP ID とオリジンが適切に設定されています
- **設定内容**:
  - RP ID: `mukwty.com`
  - RP Origins: `https://nijiviewer.mukwty.com`
- **効果**:
  - フィッシング攻撃への耐性（他のドメインで Passkey は動作しない）
  - HTTPS 強制（セキュアな通信）

#### 3. 認証フローの分離

- **評価**: ✅ 良好
- **詳細**: Passkeys とメール/パスワード認証が独立して動作
- **利点**:
  - 既存ユーザーへの影響なし
  - 段階的な移行が可能
  - 認証方法の選択肢を提供

#### 4. ブラウザ互換性チェック

- **評価**: ✅ 良好
- **実装**: `lib/passkey-utils.ts` の `isWebAuthnSupported()`
- **効果**:
  - 未サポートブラウザでエラーを防止
  - 適切なフォールバック（メール/パスワード認証）

#### 5. エラーハンドリング

- **評価**: ✅ 良好
- **実装**: `getPasskeyErrorMessage()` による包括的なエラー処理
- **カバー範囲**:
  - ユーザーキャンセル
  - ネットワークエラー
  - 無効な Passkey
  - タイムアウト
  - RP ID ミスマッチ

#### 6. クライアントサイド検証

- **評価**: ✅ 良好
- **実装**:
  - WebAuthn サポートチェック
  - エラー状態の適切な表示
  - ユーザーへの明確なフィードバック

## 脅威モデル分析

### 1. フィッシング攻撃

- **リスク**: 低
- **理由**: WebAuthn は RP ID を検証するため、偽サイトでは Passkey が動作しない
- **追加対策**: HTTPS 強制、RP ID の厳格な設定

### 2. 中間者攻撃 (MITM)

- **リスク**: 低
- **理由**: HTTPS 通信、Challenge-Response 認証
- **追加対策**: Supabase が TLS 証明書を管理

### 3. デバイス盗難・紛失

- **リスク**: 中
- **理由**: デバイス保護が必要（PIN、生体認証）
- **ユーザー責任**: デバイスレベルのセキュリティ設定
- **対策**:
  - 複数デバイスへの Passkey 登録を推奨
  - メール/パスワード認証のバックアップ維持

### 4. アカウント乗っ取り

- **リスク**: 低
- **理由**: Passkey は秘密鍵がデバイス内に保存され、サーバーに送信されない
- **追加対策**: Supabase のセッション管理、RLS ポリシー

### 5. リプレイ攻撃

- **リスク**: 低
- **理由**: WebAuthn は Challenge-Response 方式で、毎回新しい Challenge を生成
- **実装**: Supabase が自動的に処理

## データ保護

### クライアントサイド

- **保存データ**: なし
- **処理データ**: WebAuthn API の Challenge/Response（一時的）
- **評価**: ✅ 良好（秘密鍵はブラウザ/OS が管理）

### サーバーサイド (Supabase)

- **保存データ**:
  - Passkey ID
  - 公開鍵
  - Friendly Name (ユーザーが設定)
  - 作成日時
- **保護**:
  - RLS (Row Level Security) ポリシー
  - ユーザーは自分の Passkey のみアクセス可能
- **評価**: ✅ 良好

## コンプライアンス

### GDPR (EU 一般データ保護規則)

- **生体情報の扱い**: ✅ 適切
  - 生体情報（指紋、顔）はデバイス内で処理され、サーバーに送信されない
  - サーバーに保存されるのは公開鍵のみ

### プライバシー

- **個人情報**: Friendly Name のみ（ユーザーが任意で設定）
- **追跡**: なし（Passkey 自体に追跡情報は含まれない）
